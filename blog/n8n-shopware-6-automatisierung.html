n8n + Shopware 6: Bestellprozesse und Lagerbestände automatisieren

2026-07-08 · 8 min Lesezeit

Shopware 6 bringt eine vollstandige REST-API mit, die sich ohne Plugin-Entwicklung direkt mit n8n verbinden lasst. Fur wiederkehrende Aufgaben wie Bestellbenachrichtigungen, Lagerbestandsabgleich mit einem externen System oder automatisches Erstellen von Support-Tickets ist das oft schneller umgesetzt als ein eigenes Plugin.

1. API-Zugriff einrichten

Im Shopware-Admin unter Einstellungen → System → Integrationen einen neuen Integration-Client anlegen. Das liefert eine client_id und ein client_secret fur OAuth2 Client-Credentials-Flow -- kein Nutzer-Login notig, ideal fur Server-zu-Server-Kommunikation:

POST https://shop.example/api/oauth/token
Content-Type: application/json

{
  "grant_type": "client_credentials",
  "client_id": "SWIAXXXX",
  "client_secret": "DEIN_SECRET"
}

Die Antwort enthalt ein access_token, gultig typischerweise 10 Minuten. In n8n bildet man das am saubersten mit einem HTTP-Request-Node plus einem vorgeschalteten "Get Token"-Node ab, dessen Ergebnis in nachfolgenden Requests als Bearer-Token wiederverwendet wird.

n8n hat seit Version 1.x generische OAuth2-Credential-Unterstutzung -- damit erneuert n8n das Token automatisch, ohne dass man selbst einen Refresh-Mechanismus bauen muss.

2. Workflow: Benachrichtigung bei neuer Bestellung

Statt die API zu pollen, ist ein Webhook effizienter. Shopware kann Business Events wie checkout.order.placed per Webhook an eine n8n-Webhook-URL senden (einrichtbar unter Einstellungen → Webhooks):

Webhook-URL: https://n8n.example.com/webhook/order-placed
Event: checkout.order.placed

Der n8n-Workflow dahinter: Webhook-Trigger → Bestelldaten aus dem Payload extrahieren → Nachricht an Slack/Teams senden oder eine E-Mail-Benachrichtigung auslosen. Ublicherweise unter 10 Minuten Bauzeit fur die Grundversion.

3. Workflow: Lagerbestand mit externem System abgleichen

Fur einen zeitgesteuerten Abgleich (z.B. alle 15 Minuten) eignet sich ein Schedule-Trigger statt eines Webhooks:

Schedule-Trigger (alle 15 Min)
  → HTTP Request: GET /api/product?limit=500
  → Function-Node: Bestande mit externem System vergleichen
  → HTTP Request: PATCH /api/product/{id} (bei Abweichung)

Bei grosseren Katalogen lohnt sich Pagination uber page und limit Parameter der Shopware-API, damit ein einzelner Durchlauf nicht zu grosze Antworten auf einmal verarbeiten muss.

4. Fehlerbehandlung nicht vergessen

API-Calls schlagen gelegentlich fehl -- abgelaufenes Token, Timeout, Rate Limit. In n8n lasst sich pro Node ein "Continue On Fail" aktivieren, kombiniert mit einem nachgelagerten Error-Workflow, der bei wiederholtem Fehlschlag eine Benachrichtigung auslost statt den ganzen Workflow stumm abzubrechen.

Ohne Fehlerbehandlung merkt man einen fehlgeschlagenen Lagerbestandsabgleich oft erst, wenn bereits Bestellungen fur ausverkaufte Artikel reinkommen.

5. Self-hosted vs. n8n Cloud

Fur Shops mit sensiblen Daten spricht viel fur eine selbst gehostete n8n-Instanz (z.B. per Docker neben dem Shop), da Bestell- und Kundendaten dann den eigenen Server nicht verlassen. n8n Cloud ist im Gegenzug schneller startklar und entfallt Wartungsaufwand fur Updates.

Fazit

Die Kombination aus Shopwares offener REST-API und n8n als visuellem Workflow-Tool deckt viele Automatisierungsaufgaben ab, fur die sich ein eigenes Plugin nicht lohnt. Fur komplexere Logik oder Performance-kritische Prozesse bleibt ein Plugin trotzdem die robustere Wahl.
