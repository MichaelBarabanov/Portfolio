Shopware 6 Performance optimieren: Caching, Elasticsearch & HTTP/2

2026-06-20 · 8 min Lesezeit

Ein langsamer Shopware-Shop kostet Conversions. Google bewertet Core Web Vitals als Rankingfaktor, und jede zusatzliche Sekunde Ladezeit senkt die Conversion Rate messbar. Dieser Artikel zeigt, wo die gunstigsten Hebel sitzen.

1. HTTP-Cache aktivieren und verstehen

Shopware 6 bringt einen eingebauten HTTP-Cache mit, der in vielen Installationen einfach ausgeschaltet ist oder falsch konfiguriert wurde. Er liegt in config/packages/shopware.yaml:

shopware:
  http_cache:
    enabled: true
    default_ttl: 7200
    stale_while_revalidate: 300
    stale_if_error: 86400

Der stale-while-revalidate-Header sorgt dafur, dass der Cache im Hintergrund erneuert wird, wahrend der Nutzer sofort eine (etwas veraltete) Antwort bekommt. Das eliminiert Cache-Miss-Spitzen.

Wichtig: Der HTTP-Cache funktioniert nur fur nicht-eingeloggte Nutzer ohne aktiven Warenkorb. Fur eingeloggte Sessions greift er nicht.

2. Elasticsearch / OpenSearch einbinden

Ab einer Produktanzahl von ca. 2.000 wird die MySQL-Volltextsuche zum Flaschenhals. Elasticsearch (oder OpenSearch) ubernimmt dann Produktlisting und Suche:

OPENSEARCH_URL=http://localhost:9200
SHOPWARE_ES_ENABLED=1
SHOPWARE_ES_INDEXING_ENABLED=1
SHOPWARE_ES_INDEX_PREFIX=sw

Nach dem Setzen der Umgebungsvariablen den initialen Index aufbauen:

bin/console es:index
bin/console es:create-alias

Danach liegt die gesamte Produktsuche auf Elasticsearch -- MySQL entlastet, Antwortzeiten fur Suchanfragen sinken von oft 800 ms auf unter 50 ms.

3. Datenbankabfragen profilieren

Slow Queries sind der haufigste Grund fur hohe TTFB. Mit dem Symfony Profiler (nur im Dev-Modus) sieht man exakt, welche Abfragen wie lange dauern:

APP_ENV=dev
APP_DEBUG=true

Im Produktionsbetrieb hilft das MySQL Slow Query Log:

slow_query_log = ON
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 0.5

Typische Kandidaten: fehlende Indizes auf product.product_number, order.order_number, oder N+1-Probleme in Custom Plugins durch ungecachte Repository-Calls in Loops.

4. Assets mit HTTP/2 und Kompression ausliefern

HTTP/2 Multiplexing und Brotli-Kompression lassen sich auf Nginx-Ebene aktivieren:

server {
    listen 443 ssl http2;
    brotli on;
    brotli_comp_level 6;
    brotli_types text/css application/javascript image/svg+xml;

    gzip on;
    gzip_vary on;
    gzip_types text/css application/javascript;
}

5. Theme kompilieren und Assets minimieren

In der Produktion immer das Theme mit aktiviertem Minifier bauen:

bin/console theme:compile --active-only
bin/console assets:install

Das reduziert JS- und CSS-Dateigroßen typischerweise um 30-50 %.

Fazit

Die gunstigsten Gewinne kommen in dieser Reihenfolge: HTTP-Cache korrekt konfigurieren, Theme minifiziert deployen, dann Elasticsearch fur großere Kataloge. Datenbankoptimierungen lohnen sich besonders bei wachsenden Shops mit vielen Bestellungen.
