Bot-Traffic in Shopware 6 mit Cloudflare eliminieren

2026-05-28 · 6 min Lesezeit

Aggressiver Bot-Traffic ist fur Shopware-Shops ein echtes Problem: Scraper laden Produktkataloge, Credential-Stuffing-Bots probieren gestohlene Passworter durch, und simple Crawler generieren Last ohne irgendeinen Mehrwert. Cloudflare bietet mehrere Schutzschichten die sich gut kombinieren lassen.

1. Bot Fight Mode aktivieren

Der einfachste erste Schritt: Im Cloudflare-Dashboard unter Security → Bots den Bot Fight Mode einschalten. Er blockiert bekannte Bad Bots anhand von Cloudflares globaler Threat Intelligence ohne dass man selbst Regeln schreiben muss.

Free-Plan: Bot Fight Mode. Pro-Plan und hoher: Super Bot Fight Mode mit feinerer Kontrolle uber automatisierte und verifizierte Bots.

2. WAF-Regel fur den Shopware-Admin

Das Shopware-Admin-Panel unter /admin sollte nur aus bekannten IP-Ranges erreichbar sein. Mit einer Cloudflare WAF Custom Rule:

(http.request.uri.path contains "/admin") and
(not ip.src in {203.0.113.10 203.0.113.20})

Aktion: Block. Damit ist das Admin-Panel fur alle IPs gesperrt, die nicht explizit erlaubt sind.

IP-Allowlist regelmaszig aktualisieren, z.B. wenn das Buro oder VPN-IP wechselt -- sonst sperrt man sich selbst aus.

3. Rate Limiting fur Login und Checkout

Credential-Stuffing-Angriffe treffen typischerweise /account/login und /checkout/confirm mit hunderten Anfragen pro Minute. Cloudflare Rate Limiting (ab Pro) begrenzt das:

Matching criteria:
  URI Path: /account/login
  Method: POST

Rate: 5 requests per 1 minute per IP
Action: Block for 10 minutes

Dasselbe fur /checkout/confirm mit etwas groszerem Limit (z.B. 10/min) da echte Nutzer gelegentlich mehrfach bestatigen.

4. Turnstile statt Google reCAPTCHA

Shopware nutzt standardmaszig kein CAPTCHA. Wer es einbauen will, sollte Cloudflare Turnstile statt reCAPTCHA verwenden -- Turnstile sendet keine Daten an Google, ist DSGVO-freundlicher und unsichtbar fur den Nutzer.

Integration in ein Shopware-Plugin: Turnstile-Script im Storefront-Template einbinden, Token im POST-Body mitschicken, serverseitig via Cloudflare API verifizieren:

POST https://challenges.cloudflare.com/turnstile/v0/siteverify
{
  "secret": "DEIN_SECRET_KEY",
  "response": "$TOKEN_AUS_DEM_FORMULAR",
  "remoteip": "$KUNDE_IP"
}

Bei "success": true in der Antwort die Aktion durchfuhren, sonst ablehnen.

5. Bekannte Scraper-User-Agents blocken

Viele einfache Scraper verraten sich durch ihren User-Agent. Eine WAF-Regel faugt die haufigsten:

(http.user_agent contains "python-requests") or
(http.user_agent contains "Go-http-client") or
(http.user_agent contains "scrapy") or
(http.user_agent contains "curl") or
(http.user_agent eq "")

Aktion: Challenge (nicht Block), damit legitime Monitoring-Tools nicht versehentlich blockiert werden.

6. Ergebnisse messen

Im Cloudflare-Dashboard unter Analytics → Security sieht man exakt, wie viele Anfragen welche Regel getriggert hat. Nach einer Woche ist gut abzulesen ob das Rate-Limit zu restriktiv ist oder Scraper weiterhin durchkommen.

Typische Ergebnisse nach einer vollstandigen Konfiguration: 80-95 % des Bot-Traffics werden geblockt, Serverauslastung und Bandbreite sinken spurbar, und das MySQL Slow Query Log wird deutlich dunner.

Fazit

Die Kombination aus Bot Fight Mode, Admin-IP-Allowlist und Rate Limiting auf Login/Checkout deckt die meisten Angriffsvektoren ab. Der Aufwand fur die initiale Konfiguration liegt bei unter einer Stunde; danach lauft der Schutz wartungsarm im Hintergrund.
