BotProtection

Application-Layer-Schutz für Formulare in Shopware 6 – dort, wo Netzwerkschutz wie Cloudflare nicht hinsieht. Honeypot, Rate-Limiting und Time-Validation, mit Admin-Dashboard und optionaler KI-Bedrohungsanalyse.

// features

Was es kann.

  • Honeypot-Feld – unsichtbar per Off-Screen-Positionierung platziert, nicht per display:none, das einfache Bots erkennen
  • Rate-Limiting pro Endpoint – prüft IP und Session kombiniert, sodass ein einzelner Client blockiert wird, ohne ein ganzes Firmennetzwerk zu bestrafen
  • Time-based Form-Validation – blockiert Formulare, die schneller abgeschickt werden als ein konfigurierbarer Schwellenwert; das Zeit-Token ist HMAC-signiert und braucht keine Session
  • Admin-Dashboard – Statistiken, 14-Tage-Chart, die letzten blockierten Versuche und die komplette Plugin-Konfiguration an einem Ort
  • Optionale KI-Bedrohungsanalyse – Ein-Klick-Zusammenfassung der letzten 7 Tage, powered by Claude; übertragen werden nur aggregierte Zahlen, nie IPs oder User-Agents
  • DSGVO-konformes Logging – IP-Adressen werden nach 7 Tagen entfernt, ganze Einträge nach 30 Tagen gelöscht (beides konfigurierbar); Session-IDs nur als SHA-256-Hash
// was es nicht macht

Was es bewusst nicht macht.

  • Kein CAPTCHA und keine Drittanbieter-Dienste – alles läuft selbst-gehostet innerhalb von Shopware
  • Keine Firewall oder Netzwerk-Level-IP-Sperre – ausschließlich Application-Layer
  • Blockierte Anfragen bekommen eine saubere Fehlermeldung (JSON-Alert oder Flash-Message + Redirect), nie einen harten 403

// installation

Installation.

Plugin nach custom/plugins/ kopieren, dann installieren und aktivieren:

michael@shop:~
$ cp -r Shopware---BotProtection custom/plugins/MichaBotProtection
$ bin/console plugin:refresh
$ bin/console plugin:install --activate MichaBotProtection
$ bin/console cache:clear
// architektur

Architektur.

  • Subscriber/FormProtectionSubscriber – hängt sich nach dem Routing in den Kernel-Request und prüft die vier geschützten Routen
  • Service/HoneypotValidator, TimeValidator, RateLimiter – reine, unit-getestete Erkennungslogik
  • Service/BlockedAttemptLogger – schreibt blockierte Versuche über das DAL-Repository
  • Core/Content/BlockedAttempt – DAL-Entity für die Log-Tabelle
  • ScheduledTask/CleanupTaskHandler – tägliche DSGVO-Bereinigung
  • Service/AiThreatAnalyzer + Controller/ThreatAnalysisController – optionale KI-Analyse über die Admin-API
// verifiziert

Nicht nur getestet – live geprüft.

BotProtection wurde end-to-end gegen eine laufende Shopware 6.7 Instanz getestet, nicht nur isoliert unit-getestet. Drei Angriffsszenarien wurden live ausgelöst und erschienen innerhalb von Sekunden im Admin-Dashboard.

20 Tests · 126 Assertions PHPStan Level 8 0 Errors
michael@shop:~
1. Honeypot – verstecktes Feld gefüllt (Kontaktformular)
$ curl -X POST https://shop.example/form/contact -d "micha_bp_website=http://spam-bot.example" -d "email=bot@spam.example"
[{"type":"danger","alert":"Your request could not be processed."}]
2. Time-Validation – kein signiertes Token (Login)
$ curl -X POST https://shop.example/account/login -d "email=bot2@spam.example" -d "password=irrelevant123"
status: 302 → redirect mit Flash-Message
3. Rate-Limit – 6 Requests, Limit ist 5 (Newsletter)
Request 1–5: "type":"success"
Request 6: "type":"danger" ← blockiert, Limit überschritten

Alle drei Versuche erscheinen sofort unter Settings → Bot Protection – mit korrektem Grund, Zeitstempel und IP.
